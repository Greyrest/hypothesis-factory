# Фабрика гипотез — микросервисный MVP

Интеллектуальная система **генерации и приоритизации технологических гипотез**
снижения потерь цветных металлов с хвостами флотации.
Хакатон Норникеля, «Задача 1. Фабрика гипотез».

Вход — отчёт института о хвостах (`Хвосты *.xlsx`). Выход — ранжированные карточки
гипотез с обоснованием и цитатами, оценкой эффекта на KPI, рисками и дорожной
картой проверки + веб-дашборд с графом знаний и обратной связью эксперта.

## Архитектура

```
frontend (SPA, :8088)
      │ HTTP
      ▼
gateway (FastAPI, :8000)  ← публичный API, оркестрация, экспорт, оценка
      │
      ├─▶ ingestion  (:8001)  парсер xlsx-отчётов; устойчив к #REF!, пропускам,
      │                       1-2 потокам, сводным блокам
      ├─▶ knowledge  (:8002)  база знаний: справка института (чанки с цитированием),
      │                       каталог 26 практик 4 фабрик, 7 правил физики
      │                       обогащения; лексический retrieval
      ├─▶ generation (:8003)  диагностика потерь (сигналы с тоннами) + rule-based
      │         │             генерация + ранжирование + слияние с LLM-ответом
      │         └─▶ llm (:8004)  единственная точка работы с нейронкой:
      │                          провайдеры anthropic | mock | none за общим
      │                          интерфейсом и единой JSON-схемой ответа
      └─ libs/contracts        общие pydantic-модели (карточка гипотезы и др.)
```

**Замена нейронки** — главное свойство раскладки: провайдер и модель задаются
env-переменными `LLM_PROVIDER` / `LLM_MODEL` только у сервиса `llm`
(перезапускается один контейнер). Остальные сервисы знают лишь единую схему
structured-ответа. При недоступности LLM (нет ключа/сети) конвейер штатно
работает в rule-based режиме — требование локального контура из ТЗ.

```
solution/
├── services/
│   ├── gateway/       # публичный REST API + оркестрация + экспорт + оценка
│   ├── ingestion/     # xlsx -> структурированный JSON
│   ├── knowledge/     # база знаний + retrieval (docx/pdf из DATA_DIR)
│   ├── generation/    # диагностика + генерация + ранжирование
│   └── llm/           # провайдеры LLM за общим интерфейсом
├── libs/contracts/    # hf_contracts: общие модели данных
├── frontend/          # SPA-дашборд (vanilla JS + vis-network)
├── openapi/           # экспортированные OpenAPI-спеки всех сервисов
├── scripts/           # run_local.sh, export_openapi.py
└── docker-compose.yml
```

## Запуск

### Docker

```bash
cd solution
ANTHROPIC_API_KEY=sk-... docker compose up --build   # с LLM
LLM_PROVIDER=none docker compose up --build          # оффлайн rule-based
```

Дашборд: http://localhost:8088 · API: http://localhost:8000/docs

### Локально без Docker

```bash
bash scripts/run_local.sh            # venv -> deps -> 5 сервисов + статика
bash scripts/run_local.sh --no-llm   # принудительно без LLM
```

Затем на дашборде «▶ Запустить конвейер» (или `curl -X POST
localhost:8000/api/v1/runs`). Требуется Python 3.10+; путь к данным —
env `DATA_DIR` (по умолчанию `../Задача 1. Фабрика гипотез/Задача 1`).

## API (gateway)

OpenAPI: `/docs`, `/openapi.json`; снапшоты всех сервисов — в `openapi/*.json`
(обновление: `python scripts/export_openapi.py`).

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/v1/health` | Статус gateway + всех сервисов |
| POST | `/api/v1/runs?use_llm=` | Прогон всех отчётов из DATA_DIR; body (опц.) — KPI и ограничения |
| POST | `/api/v1/reports?use_llm=` | Конвейер для загруженного xlsx + интерактивные KPI/ограничения |
| GET | `/api/v1/plants` | Список фабрик со сводками |
| GET | `/api/v1/plants/{plant}` | Полный результат (гипотезы, находки, ячейки) |
| GET | `/api/v1/plants/{plant}/hypotheses` | Карточки гипотез |
| POST | `/api/v1/plants/{plant}/hypotheses` | Произвольная гипотеза эксперта: авто-категория, адресуемый металл, обоснование, ранжирование |
| PATCH | `/api/v1/plants/{plant}/hypotheses/{id}` | Подтвердить / отклонить гипотезу |
| DELETE | `/api/v1/plants/{plant}/hypotheses/{id}` | Удалить гипотезу |
| GET | `/api/v1/plants/{plant}/graph` | Граф знаний (класс -> диагноз -> гипотеза -> KPI) с правками |
| POST | `/api/v1/plants/{plant}/graph/patch` | Корректировка графа: удаление узлов/рёбер, новые связи |
| DELETE | `/api/v1/plants/{plant}/graph/patch` | Сброс правок графа |
| GET/PUT | `/api/v1/weights` | Веса ранжирования эксперта -> переранжирование |
| GET | `/api/v1/plants/{plant}/whatif?signal=&reduction_pct=` | Контрфактуальный анализ «что-если» по ячейкам отчёта |
| POST | `/api/v1/feedback` | 👍/👎 эксперта по категории -> переранжирование |
| GET | `/api/v1/evaluation` | Leave-one-out покрытие тем эталона |
| GET | `/api/v1/plants/{plant}/export?fmt=json\|csv\|md\|docx&lang=ru\|en\|zh` | Экспорт; en/zh — перевод через LLM-слой |

Безопасность: env `HF_API_KEY` включает авторизацию — все запросы (кроме
health) требуют заголовок `X-API-Key`; дашборд запросит ключ при 401.

## Соответствие ТЗ

| Требование ТЗ | Реализация |
|---|---|
| Приём структурированных данных | ingestion: парсер xlsx с обработкой неполных/битых данных (#REF! у КГМК восстанавливается из форм) |
| Парсинг текстовой литературы | knowledge: справка и эталонные docx -> чанки/каталог; PDF-учебники — метазаписи источников |
| Извлечение сущностей и связей | потоки, классы, минеральные формы, извлекаемость; граф знаний на дашборде |
| Паттерны и пробелы | сигналы диагностики; «пробельные» гипотезы для сигналов без практик в каталоге |
| Аналогии | перенос практик между фабриками (каталог собран со всех 4 примеров) |
| Ранжирование | 0.40·эффект + 0.30·реализуемость + 0.20·(1-риск) + 0.10·новизна + фидбэк + штраф за однотипность |
| Обоснование + цитирование | evidence: цифры отчёта + справка + правила с источниками (Абрамов и др.) |
| Механизм, KPI, риски, дорожная карта | в каждой карточке; KPI — диапазон тонн с явным допущением |
| Визуализация | граф класс -> диагноз -> гипотеза (vis-network); heatmap потерь |
| Экспорт | JSON, CSV, Markdown, DOCX (PDF — печать MD/DOCX) |
| Настройка весов ранжирования | панель «Настройки эксперта» на дашборде + PUT /weights |
| Контрфактуальный анализ | блок «Что-если»: устранение N% потерь сигнала -> эффект на KPI |
| Мультиязычность | RU нативно; EN/CN — перевод отчёта через LLM-слой (fmt=md&lang=en\|zh) |
| Безопасность | локальный контур, HF_API_KEY (X-API-Key), LLM отключаем |
| Метаданные источников | автор/дата docx-документов в базе знаний |
| Интерактивный ввод задачи | KPI + ограничения задаются в форме дашборда, попадают в контекст LLM и карточку проекта |
| Гипотезы эксперта | любая гипотеза свободным текстом: авто-категория, адресуемый металл по ячейкам, обоснование, общее ранжирование (status: expert_added) |
| Корректировка графа | эксперт удаляет узлы/рёбра и добавляет связи; правки хранятся отдельно и переживают повторную генерацию |
| Обратная связь эксперта | 👍/👎 по категории + подтвердить/отклонить карточку -> мгновенное переранжирование |
| Интерпретируемость | каждое число прослеживается до ячейки отчёта; допущения выписаны |
| Локальное развёртывание | LLM_PROVIDER=none — полностью оффлайн; docker compose в закрытом контуре |
| Сменная LLM | отдельный llm-сервис: провайдер/модель через env, mock-провайдер для тестов |

## Тесты

```bash
.venv/bin/pip install pytest && .venv/bin/python -m pytest tests -q   # 28 тестов
```

Каждый сервис тестируется изолированно (межсервисные вызовы замоканы):
парсер и восстановление #REF! из форм, сигналы диагностики, полнота карточек,
физический контроль (адресуемый металл ≤ потерь), ранжирование с фидбэком и
штрафом за однотипность, произвольные гипотезы эксперта, сменные LLM-провайдеры
и деградация без ключа, retrieval с цитированием, API-сценарии gateway
(проект, гипотезы, граф с правками, экспорт на лету).

## Проверка качества

Leave-one-out по темам против эталонных мозговых штурмов экспертов
(`GET /api/v1/evaluation`): покрытие тем эталона — **КГМК 67%, НОФ Вкр 100%,
НОФ мед 83%, ТОФ 100%** (подтверждено сквозным прогоном микросервисной версии).

Физический контроль: адресуемый металл гипотезы = объединение ячеек отчёта
(поток × класс × форма), поэтому никогда не превышает фактические потери.

## Ограничения и развитие

- Оценка эффекта — допущение по категориям (диапазон), требует калибровки на
  исторических парах «мероприятие -> эффект».
- PDF-учебники индексируются как источники, но не как полнотекстовый RAG
  (следующий шаг: векторный индекс в knowledge-сервисе).
- Хранение результатов — файловое (state.json); при масштабировании — БД
  за gateway.
- Универсальное ядро + доменные адаптеры (см. `docs/TZ_modularization.md`) —
  целевая эволюция: доменная логика generation-сервиса выносится в адаптер
  `mining_flotation`.
