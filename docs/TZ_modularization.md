## 1. Общие сведения

### 1.1 Назначение системы

«Фабрика гипотез» — интеллектуальная система поддержки старта научно-исследовательских и технологических проектов. Система помогает исследователям, инженерам и R&D-менеджерам формировать, обосновывать, сравнивать и приоритизировать проверяемые гипотезы на основе целевого KPI, ограничений, научно-технической базы знаний и исторических экспериментальных данных.

Система предназначена для работы в разных предметных областях: материаловедение, металлургия, полимеры, композиты, покрытия, катализаторы, химические процессы, обогащение руд и другие R&D-направления.

Ключевой принцип архитектуры: **ядро системы является доменно-независимым**, а предметная специфика подключается через отдельные доменные адаптеры.

### 1.2 Основная задача

Система должна отвечать на вопрос:

> Какие проверяемые действия могут улучшить заданный KPI с учётом доступных данных, ограничений, источников знаний, рисков и ожидаемой ценности?

Примеры входных задач:

- повысить жаропрочность сплава на 15%;
- снизить себестоимость шихты без потери прочности;
- повысить коррозионную стойкость покрытия;
- увеличить удельную прочность композита;
- снизить пористость материала;
- повысить проводимость;
- снизить потери полезного компонента в технологическом процессе.

### 1.3 Входные данные

Система принимает:

- целевой KPI или технологическую проблему;
- ограничения проекта: сырьё, бюджет, оборудование, нормативные требования, сроки, доступные методы анализа;
- научные публикации;
- патенты;
- внутренние отчёты;
- исторические эксперименты;
- таблицы с составами, режимами, параметрами и результатами испытаний;
- учебники, справочники и методические документы;
- экспертные комментарии и ранее сформированные гипотезы.

Поддерживаемые форматы:

- PDF;
- DOCX;
- TXT / Markdown;
- CSV;
- XLSX;
- JSON;
- подключение к базе данных через API или коннектор — опционально.

### 1.4 Выходные данные

Система формирует:

- ранжированный список проверяемых гипотез;
- карточку каждой гипотезы;
- механизм влияния на KPI;
- обоснование с привязкой к источникам;
- оценку новизны;
- оценку рисков;
- оценку реализуемости;
- оценку потенциальной ценности;
- оценку неопределённости;
- дорожную карту проверки;
- критерии успеха и провала;
- граф связей между фактами, источниками, механизмами и гипотезами;
- отчёт для исследователей и R&D-менеджеров;
- экспорт в JSON, CSV, Markdown, DOCX/PDF;
- опционально — экспорт задач в Jira, YouTrack или другую систему управления работами.

## 2. Предметная модель

### 2.1 Главная сущность

Главная сущность системы — **гипотеза**.

Универсальная форма гипотезы:

```text
Если изменить фактор X в условиях Y,
то целевой KPI Z изменится на величину Δ,
потому что сработает механизм M.
```

Примеры:

```text
Если добавить 0.2–0.3% Nb в сплав X и применить режим отжига Y,
то жаропрочность может повыситься на 10–15%,
потому что сформируются дисперсные карбиды, тормозящие движение дислокаций.
```

```text
Если заменить часть дорогостоящего компонента шихты на доступный аналог
при сохранении режима спекания,
то себестоимость может снизиться без потери прочности,
потому что сохраняется фазовый состав и плотность материала.
```

```text
Если изменить реагентный режим флотации для тонких классов,
то потери Ni/Cu могут снизиться,
потому что повысится извлечение раскрытых частиц.
```

### 2.2 Универсальные сущности

| Сущность | Назначение |
|---|---|
| `Project` | R&D-задача, в рамках которой генерируются гипотезы |
| `ProjectGoal` | Целевой KPI, технологическая проблема, желаемое изменение |
| `Constraint` | Ограничение проекта: сырьё, бюджет, оборудование, нормативы, сроки |
| `SourceDocument` | Документ-источник: статья, патент, отчёт, учебник, справочник |
| `Experiment` | Исторический или планируемый эксперимент |
| `Fact` | Извлечённый факт из текста, таблицы или базы данных |
| `Entity` | Материал, процесс, свойство, параметр, оборудование, реагент и т.д. |
| `Relation` | Связь между сущностями: влияет на, улучшает, ухудшает, требует, ограничивает |
| `Problem` | Выявленная проблема, пробел, аномалия или зона улучшения |
| `Hypothesis` | Проверяемое предположение с механизмом и ожидаемым эффектом |
| `Evidence` | Основание гипотезы: источник, факт, аналогия, правило, модель |
| `ExpectedEffect` | Оценка ожидаемого влияния на KPI |
| `ExperimentPlan` | План проверки гипотезы |
| `Score` | Оценка по критерию: эффект, новизна, риск, реализуемость, стоимость |
| `ExpertFeedback` | Оценка, комментарий или корректировка эксперта |
| `Trace` | Цепочка обоснования от гипотезы к источникам и данным |

### 2.3 Доменные сущности

Доменные сущности описываются в адаптерах и не должны быть зашиты в универсальное ядро.

Пример для материаловедения:

| Тип | Примеры |
|---|---|
| Материал | сплав, полимер, композит, покрытие |
| Свойство | жаропрочность, предел прочности, твёрдость, вязкость, проводимость |
| Процесс | отжиг, закалка, спекание, легирование, напыление |
| Параметр | температура, время выдержки, скорость охлаждения, концентрация |
| Механизм | дисперсное упрочнение, измельчение зерна, снижение пористости, фазовый переход |

Пример для обогащения руд:

| Тип | Примеры |
|---|---|
| Поток | хвосты, концентрат, промежуточный продукт |
| Класс крупности | `+125`, `-45+20`, `-10` |
| Минеральная форма | раскрытая, закрытая, сросток, примесь |
| KPI | снижение потерь полезного компонента |
| Механизм | доизмельчение, классификация, изменение реагентного режима |

## 3. Область применения и демонстрационные домены

### 3.1 Универсальный контур

Система проектируется как универсальная платформа для R&D-гипотез. Она не должна зависеть от одной отрасли, одного формата отчёта или одного типа KPI.

Базовый конвейер одинаков для всех направлений:

```text
KPI + ограничения + данные + база знаний
→ извлечение фактов и связей
→ доменная интерпретация
→ retrieval и граф знаний
→ генерация гипотез
→ оценка и ранжирование
→ экспертная валидация
→ экспорт и дорожная карта проверки
```

### 3.2 MVP-домены

Для MVP рекомендуется реализовать два демонстрационных домена:

1. **`materials_science`** — основной демонстрационный домен под исходную постановку R&D-задачи.
2. **`mining_flotation`** — дополнительный демонстрационный домен для кейса снижения потерь полезных компонентов в хвостах флотации.

Допускается добавить упрощённые адаптеры:

- `polymers`;
- `composites`;
- `coatings`;
- `catalysts`.

Главное требование: каждый домен подключается через адаптер и не меняет ядро системы.

## 4. Целевая архитектура

### 4.1 Общая схема

```text
Пользователь
  ├─ целевой KPI
  ├─ ограничения
  ├─ предметная область
  └─ документы / данные
        │
        ▼
Backend API
        │
        ▼
Data Ingestion

        │
        ▼

        │
        ▼
Domain Adapter

        │
        ▼

  источники, факты, эксперименты, связи, аналогии
        │
        ▼

        ▼
```

## 6. Доменные адаптеры

### 6.1 Назначение

Доменный адаптер содержит предметную логику конкретной области:

- словарь сущностей;
- правила диагностики;
- шаблоны гипотез;
- допустимые параметры и ограничения;
- KPI-метрики;
- формулы оценки эффекта;
- критерии рисков;
- типовые дорожные карты проверки;
- правила извлечения данных из доменных форматов.

### 6.2 Интерфейс адаптера

```python
class DomainAdapter(Protocol):
    domain_id: str
    title: str

    def parse_inputs(self, files: list[InputFile]) -> list[Fact]:
        ...

    def extract_entities(self, facts: list[Fact]) -> list[Entity]:
        ...

    def diagnose(
        self,
        goal: ProjectGoal,
        facts: list[Fact],
        constraints: list[Constraint],
    ) -> list[Problem]:
        ...

    def build_generation_context(
        self,
        goal: ProjectGoal,
        problems: list[Problem],
        facts: list[Fact],
        knowledge: list[Evidence],
    ) -> GenerationContext:
        ...

    def generate_rule_based(
        self,
        context: GenerationContext,
    ) -> list[HypothesisDraft]:
        ...

    def estimate_effect(
        self,
        hypothesis: HypothesisDraft,
        facts: list[Fact],
    ) -> ExpectedEffect:
        ...

    def validation_plan(
        self,
        hypothesis: HypothesisDraft,
    ) -> ExperimentPlan:
        ...
```

### 6.3 Адаптер `materials_science`

Основной демонстрационный адаптер.

Примеры KPI:

- повысить жаропрочность;
- увеличить предел прочности;
- снизить себестоимость шихты;
- повысить коррозионную стойкость;
- снизить пористость;
- повысить проводимость;
- увеличить ресурс покрытия.

Примеры направлений гипотез:

- изменение легирования;
- изменение режима термообработки;
- изменение температуры синтеза;
- изменение скорости охлаждения;
- замена сырья;
- подбор связующего;
- изменение размера частиц;
- изменение доли наполнителя;
- изменение режима нанесения покрытия;
- изменение атмосферы обработки.

Пример карточки гипотезы:

```json
{
  "title": "Микролегирование ниобием для повышения жаропрочности",
  "hypothesis": "Если добавить 0.2–0.3% Nb в сплав X и применить режим отжига Y, то жаропрочность может повыситься на 10–15%.",
  "mechanism": "Nb способствует формированию дисперсных карбидов, которые тормозят движение дислокаций и стабилизируют структуру при высоких температурах.",
  "target_kpi": "жаропрочность",
  "expected_effect": "10–15%",
  "evidence": [
    "Публикация A: описывает влияние Nb на дисперсное упрочнение",
    "Эксперимент B: похожий состав показал рост прочности при 800°C"
  ],
  "risks": [
    "рост стоимости легирования",
    "риск ухудшения пластичности",
    "необходимость уточнения режима термообработки"
  ],
  "validation_plan": [
    "Изготовить 3 состава с 0.1%, 0.2%, 0.3% Nb",
    "Провести термообработку по режимам Y1/Y2",
    "Измерить жаропрочность и пластичность",
    "Сравнить с базовым сплавом"
  ]
}
```

### 6.4 Адаптер `mining_flotation`

Демонстрационный адаптер для задач обогащения руд.

Он отвечает за:

- чтение технологических отчётов по хвостам флотации;
- извлечение потоков, классов крупности, минеральных форм и потерь полезных компонентов;
- диагностику технологических сигналов;
- генерацию гипотез по измельчению, классификации, реагентному режиму, дофлотации и другим мероприятиям;
- расчёт адресуемого полезного компонента;
- трассировку гипотезы до конкретных данных отчёта.

Этот адаптер является одним из сценариев применения системы.

## 7. Модель гипотезы

### 7.1 Обязательная структура

```python
class Hypothesis(BaseModel):
    id: str
    rank: int | None = None

    title: str
    hypothesis: str
    domain: str
    target_kpi: str

    problem: str
    mechanism: str
    expected_effect: ExpectedEffect

    evidence: list[Evidence]
    sources: list[SourceRef]

    novelty: Score
    feasibility: Score
    risk: Score
    impact: Score
    cost: Score | None = None
    uncertainty: Score | None = None
    evidence_strength: Score | None = None
    strategic_value: Score | None = None

    constraints_matched: list[str]
    constraints_violated: list[str]

    risks: list[str]
    validation_plan: ExperimentPlan

    status: Literal[
        "rule_based",
        "rag_llm",
        "predictive_model",
        "expert_added",
        "confirmed",
        "rejected"
    ]

    trace: TraceInfo
```

### 7.2 Обязательные поля карточки

Карточка гипотезы должна содержать:

1. Название.
2. Формулировку гипотезы.
3. Целевой KPI.
4. Ожидаемый эффект.
5. Механизм влияния.
6. Обоснование.
7. Источники.
8. Оценку новизны.
9. Оценку реализуемости.
10. Оценку рисков.
11. Оценку стоимости проверки.
12. Ограничения, которым гипотеза соответствует.
13. Ограничения, которые гипотеза может нарушить.
14. План проверки.
15. Критерии успеха и провала.
16. Итоговый приоритет.
17. Экспертный фидбэк.
18. Трассировку к данным и источникам.

### 7.3 Требование к проверяемости

Гипотеза должна быть конкретной и проверяемой.

Недопустимая формулировка:

```text
Улучшить состав материала.
```

Допустимая формулировка:

```text
Если увеличить содержание Nb с 0.1% до 0.3% и провести отжиг при 800°C в течение 2 часов,
то жаропрочность сплава X может повыситься на 10–15%
за счёт формирования дисперсных карбидов.
```

## 8. Ранжирование

### 8.1 Критерии

| Критерий | Смысл |
|---|---|
| `impact` | Ожидаемое влияние на KPI |
| `novelty` | Новизна по сравнению с известными решениями |
| `feasibility` | Реализуемость в доступных условиях |
| `risk` | Технический, экономический и организационный риск |
| `cost` | Оценка ресурсов и стоимости проверки |
| `evidence_strength` | Сила обоснования |
| `uncertainty` | Неопределённость прогноза |
| `strategic_value` | Соответствие бизнес-целям |

### 8.2 Гибкие веса

Эксперт должен иметь возможность менять веса критериев:

```json
{
  "impact": 0.30,
  "novelty": 0.20,
  "feasibility": 0.20,
  "risk": 0.15,
  "cost": 0.10,
  "evidence_strength": 0.05
}
```

Для доменов могут быть заданы веса по умолчанию.

### 8.3 Базовая формула

```text
priority =
  w_impact * impact
+ w_novelty * novelty
+ w_feasibility * feasibility
+ w_evidence * evidence_strength
+ w_strategic * strategic_value
- w_risk * risk
- w_cost * cost
- w_uncertainty * uncertainty
+ expert_feedback_adjustment
```

### 8.4 Экспертная настройка

Эксперт должен иметь возможность:

- менять веса ранжирования;
- исключать направления;
- закреплять обязательные ограничения;
- понижать или повышать приоритет гипотез;
- добавлять комментарии;
- отмечать гипотезы как подтверждённые, отклонённые или требующие доработки.

## 9. Граф знаний

### 9.1 Назначение

Граф знаний обеспечивает интерпретируемость и трассировку.

Он должен показывать цепочку:

```text
Гипотеза
→ проблема / пробел / возможность
→ факты и эксперименты
→ источники
→ сущности и механизмы
→ ограничения
→ план проверки
```

### 9.2 Типы узлов

| Тип | Пример |
|---|---|
| `ProjectGoal` | повысить жаропрочность на 15% |
| `Constraint` | не использовать дефицитное сырьё |
| `Material` | сплав X |
| `Process` | отжиг, закалка, спекание |
| `Parameter` | температура, время, концентрация |
| `Property` | жаропрочность, твёрдость |
| `Experiment` | испытание образца A |
| `Fact` | прочность = 820 МПа |
| `Source` | статья, патент, отчёт |
| `Problem` | высокая пористость, низкая жаропрочность |
| `Mechanism` | дисперсное упрочнение |
| `Hypothesis` | карточка гипотезы |
| `ExperimentPlan` | план проверки |

### 9.3 Типы рёбер

| Ребро | Смысл |
|---|---|
| `TARGETS` | гипотеза влияет на KPI |
| `BASED_ON` | гипотеза основана на источнике или факте |
| `EXPLAINS` | механизм объясняет гипотезу |
| `CONSTRAINED_BY` | гипотеза ограничена условием |
| `TESTED_BY` | гипотеза проверяется экспериментом |
| `SIMILAR_TO` | связь с аналогичным известным решением |
| `AFFECTS` | параметр влияет на свойство |
| `OBSERVED_IN` | факт наблюдался в эксперименте |
| `CITES` | ссылка на источник |
| `VALIDATES` | результат подтверждает гипотезу |
| `REFUTES` | результат опровергает гипотезу |

### 9.4 Трассировка

Для каждой гипотезы должна быть доступна трассировка.

Для материаловедения:

```text
Гипотеза
→ целевой KPI
→ механизм
→ факты из экспериментов
→ статьи / патенты
→ ограничения
→ план проверки
```

Для обогащения:

```text
Гипотеза
→ технологический сигнал
→ поток
→ класс крупности
→ форма полезного компонента
→ величина потерь
→ правило / источник
```

## 10. LLM-модуль

### 10.1 Назначение

`hf_llm` отвечает только за работу с языковыми моделями:

- вызов провайдера;
- шаблоны промптов;
- структурированный вывод;
- валидацию ответа;
- обработку ошибок;
- fallback в rule-based/RAG-режим;
- телеметрию;
- кэширование.

### 10.2 Принципы

- LLM не является единственным источником истины.
- LLM не генерирует гипотезы без опоры на данные, источники и ограничения.
- Каждая гипотеза от LLM проходит валидацию схемы.
- При ошибке LLM система продолжает работу без падения.
- В локальном контуре внешние LLM-вызовы должны быть отключаемыми.
- Отправка конфиденциальных данных во внешний LLM допускается только при явном включении соответствующего режима.

### 10.3 Use cases

MVP:

1. `enhance_hypotheses` — улучшение черновиков гипотез.
2. `generate_from_context` — генерация новых гипотез из RAG-контекста.
3. `explain_hypothesis` — формирование понятного объяснения.
4. `summarize_sources` — краткое резюме источников.
5. `translate` — опционально для RU/EN/CN.

## 17. Критерии успешности

Система соответствует заданию, если она:

1. Позволяет задать произвольный R&D KPI и ограничения.
2. Работает с разнородной базой знаний.
3. Поддерживает хотя бы один полноценный доменный адаптер и архитектурно готова к новым доменам.
4. Генерирует конкретные проверяемые гипотезы.
5. Даёт обоснование с источниками.
6. Оценивает новизну, риски, реализуемость и ценность.
7. Предлагает план проверки.
8. Позволяет эксперту корректировать и переоценивать гипотезы.
9. Поддерживает трассировку от гипотезы к данным и источникам.
10. Может быть развёрнута локально и работать без внешних LLM.

## 18. Формулировка для README и презентации

> «Фабрика гипотез» — универсальная R&D-платформа, которая превращает целевой KPI, ограничения, документы и исторические эксперименты в ранжированный список проверяемых гипотез. Система извлекает факты и связи из разнородных источников, строит граф знаний, использует RAG/LLM и доменные правила, оценивает гипотезы по эффекту, новизне, рискам и реализуемости, а также формирует дорожную карту проверки. Предметные области подключаются через доменные адаптеры, поэтому решение применимо к материаловедению, металлургии, полимерам, композитам, покрытиям, катализаторам и другим R&D-направлениям.
