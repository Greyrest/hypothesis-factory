== Обзор

Система разбита на пять независимых сервисов и статический фронтенд.
Каждый сервис — отдельное FastAPI-приложение со своим Dockerfile,
зависимостями и OpenAPI-спекой; общение — только по HTTP/JSON.

[listing]
----
frontend (SPA, :8088)
      │ HTTP
      ▼
gateway (:8000)  ← публичный API, оркестрация, экспорт, оценка, auth
      │
      ├─▶ ingestion  (:8001)  xlsx → структурированный JSON
      ├─▶ knowledge  (:8002)  база знаний + retrieval
      ├─▶ generation (:8003)  диагностика + генерация + ранжирование
      │         └─▶ llm (:8004)  единственная точка работы с нейронкой
      └─ libs/contracts        общие pydantic-модели
----

== Зоны ответственности

[cols="1,3,2"]
|===
| Сервис | Ответственность | Не знает про

| `ingestion`
| Парсинг отчётов института (лист «Итог»): потоки × классы крупности ×
  минеральные формы, извлекаемый металл. Устойчив к `#REF!`, пропускам,
  1–2 потокам, сводным блокам; парсит из памяти (BytesIO).
| базу знаний, гипотезы, LLM

| `knowledge`
| База знаний: чанки справки института с цитированием и метаданными
  (автор/дата), каталог 26 практик из мозговых штурмов 4 фабрик,
  7 правил физики обогащения с привязкой к литературе; лексический retrieval.
| парсинг xlsx, ранжирование

| `generation`
| Правила диагностики (сигналы с тоннами по ячейкам отчёта), rule-based
  генерация «сигналы × каталог», обработка произвольных гипотез эксперта,
  контрфактуальный анализ, ранжирование с настраиваемыми весами.
| формат xlsx, конкретную LLM

| `llm`
| Абстракция провайдеров (`anthropic`/`mock`/`none`) за единой JSON-схемой
  structured-вывода; use cases: enhance (полировка + новые гипотезы),
  translate (EN/CN). Недоступность LLM — штатный ответ `null`.
| диагностику, ранжирование, хранение

| `gateway`
| Публичный REST API, оркестрация конвейера, состояние (память + снапшот
  `state.json`), граф знаний с правками эксперта, экспорт JSON/CSV/MD/DOCX
  на лету, leave-one-out оценка, авторизация по API-ключу.
| физику обогащения
|===

== Поток данных

. Пользователь задаёт KPI/ограничения и запускает конвейер (или загружает свой отчёт).
. `gateway` читает xlsx и передаёт байты в `ingestion` → структурированный JSON.
. `gateway` вызывает `generation`; тот берёт базу у `knowledge`,
  строит диагностику (сигналы: недоизмельчение, шламовые потери,
  недофлотация, проскок классификации, пирротин), генерирует карточки
  из каталога практик и «пробельные» гипотезы.
. При включённой LLM `generation` шлёт черновики в `llm`; structured-ответ
  сливается с rule-based карточками (черновики не теряются).
. Результат ранжируется: `w·эффект + w·реализуемость + w·(1−риск) + w·новизна`
  + фидбэк по категориям + статус валидации + штраф за однотипность.
. `gateway` хранит результат, строит граф знаний
  (класс → диагноз → гипотеза → KPI) и применяет правки эксперта.

== Замена нейронки

Ключевое проектное решение: вся работа с LLM изолирована в сервисе `llm`.

* Интерфейс — `services/llm/app/providers/base.py` (`enhance`, `translate`).
* Схема ответа (`schema.py`) одинакова для всех провайдеров — генератор
  не знает, какая модель за ней стоит.
* Смена модели: `LLM_MODEL=... make restart-llm` — перезапускается один
  контейнер, остальные сервисы не трогаются.
* Новый бэкенд (локальная модель, другой вендор) — один класс-провайдер
  и ветка в `providers/__init__.py`.

== Состояние и персистентность

Всё состояние gateway (результаты, правки графа, фидбэк, веса, оценка)
живёт в памяти; единственный файл — снапшот `output/state.json` для
переживания рестарта. `HF_PERSIST=0` отключает и его (полностью
безфайловый режим). БД в MVP нет сознательно; точка замены — `_persist()`
и `_load_state()` в `services/gateway/app/main.py`.

== Соответствие целевой архитектуре ТЗ

Целевая эволюция по `docs/TZ_modularization.md` (универсальное ядро +
доменные адаптеры): доменная логика `generation` (сигналы, категории,
uplift) выносится в адаптер `mining_flotation`, интерфейс
`DomainAdapter` позволяет подключать материаловедение и другие домены
без изменения ядра.
